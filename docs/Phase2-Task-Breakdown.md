# 第二阶段落地任务清单
> 阶段目标：完成“最小战局闭环”  
> 即实现：进入测试战局 → 玩家移动/瞄准/攻击/交互 → 搜刮基础容器 → 触发基础敌对遭遇 → 抵达撤离点 → 进入结算界面 → 返回安全区/主菜单占位流程  
> 当前日期：2026-03-12

---

# 1. 阶段目标定义

第二阶段的核心目标不是做完整游戏，而是完成一个**可重复运行的最小战局闭环（Minimum Raid Loop）**。

本阶段完成后，项目应具备以下能力：

1. 从主菜单或测试入口进入战局
2. 玩家可在战局中：
   - 移动
   - 瞄准
   - 攻击
   - 交互
3. 场景中存在至少一种可搜刮容器
4. 场景中存在至少一种基础敌人
5. 玩家可击败敌人或规避敌人
6. 玩家可进入撤离点并完成撤离
7. 战局结束后可进入结算界面
8. 结算后可返回上一级流程

---

# 2. 本阶段范围

## 2.1 必做内容
- 最小战局状态管理
- 玩家战局控制
- 基础武器与攻击
- 基础生命与受伤逻辑
- 基础敌人行为
- 基础交互系统
- 基础容器与搜刮
- 基础掉落
- 撤离区与撤离读条
- 战局结果生成
- 结果界面占位

## 2.2 可简化内容
- 武器只做 1 远程 + 1 近战占位
- 物品只做极少数测试内容
- 背包先不做完整 grid inventory，可先做最小容器/列表式中间版本，或者只做“固定槽 + 简化转移”
- 敌人 AI 只做简单追击/攻击
- 地图只做手工测试场景，不做随机生成
- 安全区只保留返回入口，不做完整功能

## 2.3 不包含内容
- 完整 grid-based inventory
- 高级部位伤害系统
- 完整任务系统
- 商人交易
- 科技树
- 农业
- 工业自动化
- 多地图随机生成
- 高级 UI 动效
- 复杂音频事件系统
- 多敌人协同 AI
- 复杂弹药/装填系统

---

# 3. 阶段成果定义

第二阶段结束时，应有一个**可演示战局样例**：

- 从菜单进入 `Raid_TestMap`
- 玩家出生在指定点
- 可使用基础武器
- 可打开一个基础容器拿到测试物品
- 可击杀一个基础敌人
- 可走到撤离点并成功撤离
- 显示结算结果
- 返回菜单或安全区占位场景

### 目标
创建战局生命周期控制器。

### 职责
- 初始化战局
- 记录玩家出生
- 跟踪战局状态
- 接收撤离成功/失败事件
- 收集结算数据

### 建议状态
- `INIT`
- `RUNNING`
- `EXTRACTING`
- `SUCCESS`
- `FAILED`
- `RESULT`

# 6. 验收标准

## 6.1 功能验收
必须满足：
- 可从菜单进入战局
- 玩家可移动、攻击、交互
- 至少一个容器可搜刮
- 至少一个敌人可追击并被击杀
- 至少一个撤离点可完成撤离
- 战局结束后可显示结果页
- 可从结果页返回主流程

## 6.2 架构验收
必须满足：
- 战局流程由 `RaidController` 管理
- 内容查询仍通过 `RegistryService`
- 输入仍通过 `InputService`
- 相机仍通过 `CameraService`
- 切场仍通过 `SceneFlowService`
- 不因赶进度绕开适配层直接乱调插件 API

## 6.3 体验验收
至少满足：
- 玩家理解“移动 → 搜刮 → 战斗 → 撤离”的流程
- 操作延迟和切场流程可接受
- HUD 与提示信息足够完成演示

# 8. 关键实现建议

## 8.1 背包不要一步到位
第二阶段不要直接上完整 grid inventory。  
建议先完成：
- 容器打开
- 物品显示
- 物品拿取
- 计入结算

等第三阶段再升级到完整 grid-based inventory。

## 8.2 武器系统先用极简模型
建议：
- 远程：hitscan + cooldown
- 近战：范围判定
- 不做弹药、换弹、卡壳

## 8.3 敌人 AI 只做单体追击
不要在本阶段做：
