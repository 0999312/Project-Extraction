# 1. 文档信息

## 1.1 项目代号
**Project Safeball**（暂定）

## 1.2 项目简介
本项目是一款 **2D 俯视角撤离射击游戏**，整体设计参考类《塔科夫》的“进入战局—搜集资源—遭遇战斗—成功撤离—返回安全区成长”的核心循环，同时引入 **任务、NPC 交易、对话、科技树、农业生产、工业自动化** 等长期成长内容。

项目采用 **高度抽象、可爱化的人类形象设计**：角色可被视为只有头、手与身体的简化形象，甚至可以直接设定为“有手、会说话的球形生物”。这种美术风格能够降低角色制作与动画成本，并增强俯视角下的辨识度。

本项目要求以仓库 **`0999312/Minecraft-Style-Framework`** 为核心制作基础，重点复用其 **注册表系统** 与 **偏数据驱动的底层设计思路**，将物品、实体、场景、任务、科技、配方等游戏内容统一纳入注册与配置体系中进行管理。

# 2. 项目目标

## 2.1 产品目标
- 构建一个可长期扩展的 **Godot 2D 撤离射击框架项目**
- 实现战局内紧张搜打撤 + 战局外成长经营的双循环体验
- 通过注册表系统统一管理内容，降低后续扩展成本
- 让基础功能具备快速开发能力，并允许复用现成模板或第三方插件
- 为后续加入更多地图、敌人、任务、基地生产线与科技系统提供架构基础

## 2.2 开发目标
- 优先完成一个具备完整闭环的 MVP
- 尽早确定注册表规范、内容标签规范、存档边界与运行时架构
- 在功能扩展前优先考虑性能与场景优化
- 尽可能采用 **“大类注册 + 标签细分”** 的方式管理物品与内容，而非大量碎片化注册表

# 4. 核心设计原则

## 4.1 使用指定 Repo 的插件/框架进行制作
项目必须基于 **`0999312/Minecraft-Style-Framework`** 的底层设计思路实施，尤其是：
- 注册表系统
- 数据驱动内容定义
- 可扩展的底层模块组织方式

如实现其他功能有必要，可引入第三方插件或模板项目，但这些应作为补充，不应取代该 Repo 在项目底层中的核心定位。

## 4.2 大类注册，细类使用标签区分
物品系统应遵循以下原则：
- 武器、药物、食物、材料等 **按大类注册**
- 不为每个小类单独拆分一个新的注册表
- 具体差异通过：
  - 标签 `tags`
  - 子类型 `subtype`
  - 能力组件 `components`
  - 行为字段 `traits`
  来表达

例如：
- 所有武器都属于 `item`
- 所有药物都属于 `item`
- `weapon/ranged`、`weapon/melee`、`medical/painkiller` 等仅作为标签，而不是单独注册表

## 4.3 内容优先数据驱动
- 地图 POI、敌人、道具、配方、科技、任务、掉落等尽量通过数据配置驱动
- 运行时代码负责解释和执行规则，而不是硬编码所有具体内容

## 4.4 先架构后堆内容
- 在大规模增加地图、敌人、物品前，先明确内容管线、对象生命周期、场景组织与性能策略
- 避免后期因早期架构混乱而重构大量系统

# 8. 美术与角色设计

## 8.1 角色风格要求
“人类”在美术上是 **高度抽象化角色**，可以视为：
- 只有头、手与身体的简版可爱形象
- 或者直接是一个“有手、会说话的球”

## 8.2 设计目标
- 降低角色动画成本
- 保持俯视角高识别度
- 允许通过帽子、武器、背包、颜色、表情区分职业与阵营
- 用简化美术承载复杂系统与硬核玩法，形成反差风格

## 8.3 角色识别层
建议通过以下元素区分角色：
- 轮廓大小
- 配件（头盔、眼镜、背包、肩饰）
- 颜色方案
- 动画状态
- 武器轮廓
- 表情与状态贴花

## 8.4 非人角色设计风格
与“球形人类”保持统一抽象语言：
- 变异体：破损球、长刺、发光眼
- 机械体：金属球、多边形球、浮空组件
- 植物体：带枝叶/孢子结构的球核
- Boss：多层核心、外部浮动模块

# 27. 架构与性能优化方案

项目必须在设计阶段优先考虑优化，而不是在内容膨胀后被动修补。

## 27.1 架构方案 A：传统场景树 + 管理器单例
### 特点
- 开发速度快
- 适合原型
- 容易上手

### 风险
- 后期耦合严重
- 单例臃肿
- 难以维护高复杂度逻辑

## 27.2 架构方案 B：数据驱动 ECS-lite / Component 系统
### 特点
- 适合大量实体
- 系统可批处理
- 扩展性强
- 性能潜力更高

### 风险
- Godot 编辑器工作流不如传统 Node 直观
- 初期设计成本高
- 团队需要更高工程能力

## 27.3 架构方案 C：混合架构（推荐）
### 思路
- 内容层：注册表 + 数据定义
- 运行层：组件化/FSM/事件系统
- 表现层：Godot Scene/Node
- 重对象 Node 化，轻对象数据化

### 推荐划分
#### Node 化对象
- 玩家
- NPC
- Boss
- 大型敌人
- 可交互设施
- 关键场景对象

#### 数据化或轻量对象
- 子弹
- 临时掉落
- 感知事件
- 临时特效
- 部分 AI 查询缓存

## 27.4 必须考虑的优化策略
### 场景层
- 地图分块加载
- 远距离实体休眠
- 室内外分层激活
- 可视区驱动更新
- 分区导航

### 实体层
- 对象池
- AI 分帧更新
- 感知系统低频化
- 避免每帧全局节点查找
- 统一事件分发

### UI 层
- 增量刷新
- Tooltip 缓存
- 列表虚拟化
- 降低复杂界面重复重绘

### 数据层
- 注册内容预加载
- 图标缓存
- 资源懒加载
- 存档分模块序列化

# 29. 风险与应对

## 29.1 主要风险
- 系统过多导致早期范围失控
- 背包系统复杂度超预期
- AI 与地图生成耦合过深
- 基地经营内容过早吞噬开发资源
- 注册表设计不统一导致扩展困难
- 存档边界定义不清导致后期兼容问题

## 29.2 应对策略
- 优先定义注册规范与标签规范
- 先做 `item / entity / quest / recipe / tech` 五大核心注册体系
- 背��系统优先做最小闭环
- 农业与工业初期仅做轻量版
- 存档结构尽早固定
- AI 与地图生成采用模块边界隔离
- 所有新系统先写入数据模型与生命周期说明，再进入实现

# 30. 成功标准

## 30.1 MVP 成功标准
- 玩家可完成一次从进入地图到成功撤离的完整流程
- 战局内可完成搜刮、战斗、负重管理、撤离
- 战局外可进行仓库整理、交易、简单任务交付
- 注册表系统已用于核心内容组织

## 30.2 中期成功标准
- 可持续添加新武器、新地图、新敌人、新任务而不需大规模重构
- 安全区成长系统开始形成长期动力
- 科技与生产链能承接战局内资源输入

## 30.3 长期成功标准
- 构建出兼具“撤离射击 + 安全区成长 + 自动化生产”的稳定框架
- 成为可长期维护与扩展的 Godot 游戏项目基础

# 31. 结论与推荐落地路线

## 31.1 推荐技术路线
- 采用 **混合架构**
- 以 **注册表系统 + 数据驱动** 为内容组织核心
- 以 **Godot Scene/Node** 负责表现层
- 使用第三方插件补齐对话、输入重绑定、AI、存档等通用功能

## 31.2 推荐玩法路线
- 以撤离射击为主玩法
- 以安全区成长为长期驱动力
- 农业与工业作为中后期科技树的重要组成部分
- 美术上坚持抽象、可爱、统一识别性的球形角色设计

## 31.3 推荐研发优先级
1. 注册表规范
2. 背包系统
3. 战斗与敌人
4. 撤离流程
5. 安全区与商人
6. 对话与任务
7. 科技树
8. 农业/工业自动化
9. 大规模内容扩展与优化
